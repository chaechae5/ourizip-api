# ADR-006: 단계별 개발 전략 — 1단계 MVP 우선 구현

- **날짜**: 2026-04-03
- **상태**: 채택됨

---

## 배경

임장 앱은 여러 기능군을 포함하고 있다. 전체를 한 번에 설계하고 구현할 경우 다음 위험이 있다.

- 사용자 검증 전 과도한 투자
- 복잡한 외부 API 의존성(공공 데이터 포털 승인 등) 대기로 인한 블로킹
- 코드베이스 복잡도 조기 증가

---

## 결정

**기능을 5단계로 분리하고, 현재는 1단계만 구현한다.**

| 단계 | 내용 | 상태 |
|---|---|---|
| **1단계 (현재)** | 지도 UI + 핀·코멘트·사진 MVP | 구현 중 |
| 2단계 | 즐겨찾기·임장 타임라인 | 미착수 |
| 3단계 | 출근지 이동시간 + 시세 시뮬레이터 ★ | 미착수 |
| 4단계 | 청약·분양 오버레이 (공공API 승인 필요) | 미착수 |
| 5단계 | 알림 고도화 | 미착수 |

---

## 이유

### 단계별 분리의 근거

- **3단계 시세 시뮬레이터**는 바이럴 포인트로 핵심 기능이지만, 1단계 핀 저장이 없으면 의미 없음. 의존성 순서를 지킴.
- **4단계 공공 API**는 데이터 포털 승인 절차가 외부 의존성. 승인 전에 코드를 짜도 통합 테스트 불가.
- MVP 출시 후 사용자 피드백에 따라 2·3단계 우선순위가 바뀔 수 있음.

### 코드에 적용하는 원칙

- **미래 단계 코드 선구현 금지**: 3단계용 `CommutingTimeService` 등을 지금 만들지 않음.
- **PostGIS 채택은 예외**: 좌표 저장 방식은 3단계 기능과 직결되므로 1단계부터 PostGIS를 사용함. 나중에 마이그레이션하면 비용이 너무 큼. (→ ADR-001, ADR-002 참고)
- **인터페이스 추상화 자제**: 현재 단일 구현체만 있는 Service에 불필요한 인터페이스 추가하지 않음.

---

## 1단계 구현 범위 (핀 기능 체크리스트)

- [x] 핀 추가 (`POST /api/v1/pins`)
- [ ] 핀 수정 (`PATCH /api/v1/pins/{id}`)
- [ ] 핀 삭제 (`DELETE /api/v1/pins/{id}`)
- [ ] 핀 목록 조회 (지도 뷰포트 기준, `GET /api/v1/pins`)
- [ ] 핀 단건 조회 (`GET /api/v1/pins/{id}`)
- [ ] 핀 검색 (`GET /api/v1/pins?keyword=`)
- [ ] 사진 업로드 Presigned URL 발급
- [ ] JWT 인증 연동

---

## 미결 사항

- 2단계 착수 시점 기준: 1단계 사용자 테스트 완료 후
- 3단계 시세 시뮬레이터 데이터 소스 결정 필요 (국토부 실거래가 API 등)
