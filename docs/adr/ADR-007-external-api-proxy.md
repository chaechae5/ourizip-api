# ADR-007: 외부 API 처리 방식 — 서버 프록시 + Redis 캐싱

- **날짜**: 2026-04-03
- **상태**: 채택됨 / Redis 미도입 (4단계 전 구현 예정)

---

## 배경

4단계에서 청약·분양 정보를 제공하기 위해 공공 데이터 포털 등 외부 API를 연동해야 한다.
클라이언트에서 직접 외부 API를 호출할지, 서버에서 프록시 처리할지를 결정해야 했다.

---

## 결정

**외부 API 호출은 서버(BE)에서만 수행하고, 응답은 Redis에 캐싱한다.**

```
[클라이언트]
  ↓ GET /api/v1/subscriptions?regionCode=11110
[서버]
  ↓ Redis 캐시 확인
  ↓ (MISS) 공공 데이터 포털 API 호출
  ↓ Redis에 TTL과 함께 저장
  ↓ 응답 반환
```

---

## 선택지 비교

| 방식 | 장점 | 단점 |
|---|---|---|
| **서버 프록시 + Redis 캐싱 (채택)** | API 키 노출 없음, 캐싱으로 호출 횟수 절감, 응답 형식 정규화 가능 | 서버 추가 구현 필요 |
| 클라이언트 직접 호출 | 서버 부담 없음 | API 키가 앱 번들에 포함 → 노출 위험, 클라이언트별 캐싱 불가 |
| 별도 외부 API 게이트웨이 서비스 | 마이크로서비스 분리 | MVP 규모에서 과도한 복잡도 |

---

## 결정 이유

### API 키 보안
- 공공 데이터 포털 API 키를 클라이언트 앱에 포함하면 디컴파일로 추출 가능.
- 서버에서만 호출하면 API 키는 환경변수(`API_KEY_OPEN_DATA`)로만 관리.

### Redis 캐싱 필요성
- 공공 API는 일별 호출 횟수 제한이 있음 (예: 일 1,000~10,000회).
- 청약 정보는 실시간성이 낮음 (하루 1회 갱신) → TTL 24시간 캐싱으로 API 호출 절감.
- 여러 사용자가 동일 지역 정보를 요청할 때 캐시 히트로 응답 속도 향상.

### 응답 정규화
- 외부 API마다 응답 스펙이 다름. 서버에서 내부 DTO로 변환해 클라이언트는 통일된 스펙만 의존.
- 외부 API 스펙 변경 시 서버만 수정하면 됨.

---

## 캐싱 설계 (계획)

| 데이터 | 캐시 키 예시 | TTL |
|---|---|---|
| 청약·분양 정보 | `subscription:region:{regionCode}` | 24시간 |
| 시세 정보 (3단계) | `price:apt:{aptCode}:{yyyyMM}` | 1시간 |

---

## 현재 상태

Redis 의존성 미추가. 4단계 착수 시 `spring-boot-starter-data-redis` 추가 예정.

---

## 미결 사항

- 공공 데이터 포털 API 승인 신청 시점
- Redis 운영 환경: AWS ElastiCache (Redis 호환) 사용 예정
- 캐시 무효화 전략: TTL 만료 방식만 사용할지, 이벤트 기반 무효화도 고려할지
